01

Claude Code 是什么

核心认知：Claude Code 不只是写代码的工具，而是一个以命令行为载体的万能 AI Agent，能自主操作你的电脑完成任何可描述的任务。
传统认知实际能力
写代码 / 编程助手数据分析、文件管理、资讯采集、修图、剪辑、日程安排……
每次对话独立通过 Skills / Hooks / Loop 实现持续自动化
你问它答你指挥，它自主执行，只汇报结果

核心类比：远古人钻木取火 vs 现代人用打火机。工具的代际差距不在于"能不能做"，而在于效率和心智负担。

启动与界面

两种启动方式

终端
# 普通模式（每次操作需确认） claude # 危险模式（全自动，无需确认）—— 推荐日常使用 claude --dangerously-skip-permissions

05

三种运行模式

Default Mode

默认模式

每次执行操作（读/写文件、运行命令）都暂停等待你确认。安全，适合学习阶段。

Plan Mode

规划模式

只读权限，先生成执行计划给你审核，确认后才写入。适合复杂任务、大范围操作。

Bypass Mode

全自动模式

跳过所有确认，直接执行。启动参数加 --dangerously-skip-permissions，推荐日常使用。

切换方式

在 Claude Code 界面内按 Shift + Tab 循环切换模式。

Plan Mode 的三个确认选项

选项含义
Yes and bypass同意计划，后续全部自动执行
Manual approve each同意计划，但每一步仍需手动确认
修改计划用自然语言说出修改意见，AI 更新后再次给你审核
Plan Mode 最适合的三类场景：①项目初期验证 AI 理解是否正确；②广度大的批量操作（移动几百个文件）；③代码 Review，先看步骤再执行。
06

CLAUDE.md——项目"宪法"

生成方式

Claude Code 内
/init # Claude 会扫描整个项目，自动生成 CLAUDE.md

作用：每次对话启动时自动加载，相当于 AI 的永久指令，优先级最高。任何时候进入新项目都应先运行 /init。

写什么

CLAUDE.md 示例
# 基本规则 - 所有回复使用中文 - 操作文件前先告诉我打算做什么 - 禁止执行 rm -rf 等危险命令 - 禁止删除任何文件，只能移动 # 详细开发规范 如需了解，请查阅 ./docs/dev-guide.md

黄金原则：渐进式披露

❌ 错误做法
CLAUDE.md 写了几千字，把所有规范、背景、注意事项全部塞进去。
✅ 正确做法
CLAUDE.md 只写核心规则（5-10 条），详细内容拆分到独立 MD 文件，CLAUDE.md 只写引用路径。
CLAUDE.md 过长的两个恶果：① 每次对话消耗大量 token；② 模型被"噪声"淹没，变得反应迟钝。
07

Agent 团队——分工协作

什么时候建 Agent

满足以下任意一条：你重复执行同一类任务，且这类任务需要很长的 Prompt 和大量规范约束。

典型 Agent 团队架构

团队结构示意
你（指挥官） ├── 产品经理 Agent → 分析需求、写 PRD、拆解任务 ├── 后端开发 Agent → 写代码、调试 ├── 前端开发 Agent → UI 实现 ├── 测试 Agent → 检查 Bug、验证功能 ├── 资讯 Agent → 每日信息采集、整理报告 └── Codex Agent → 在终端执行命令，与 Claude 自动对接

创建步骤

1
进入 Agent 管理
Claude Code 内
/agent # 选择 Create new agent
2
选择权限级别与创建方式

Location 选 Project（项目级）；创建方式选 Generate with Claude（AI 辅助生成，推荐）。

3
描述 Agent 职责
描述示例
你是一个技术合伙人 Agent。 职责：评估技术方案可行性，设计系统架构， 拆解开发任务并给出时间估算， 进行代码 Review，识别技术风险。 请用简洁专业的中文回答。
4
配置工具权限 / 模型 / 颜色

工具权限：↑↓ 导航，Enter 切换选中。模型选 Inherit 继承父级。颜色随意选，便于区分。

5
调用 Agent
对话
帮我调用技术合伙人 Agent，我要做一个新项目……

Agent 的隐藏价值：上下文隔离

子 Agent 执行大量中间操作（抓取数据、处理文件），主对话只收到一句 "任务完成"，中间过程不占用主对话的上下文，大幅节省 token。

08

四层能力架构：MCP · Skills · Hooks · Plugins

MCP

模型上下文协议

给 AI 装"手脚"——能不能做。连接外部系统，如文件读写、网络搜索、飞书操作。

Skills

技能

告诉 AI怎么做——固化工作流的 Prompt 模板，有能力后按什么步骤执行。

Hooks

钩子

控制何时做——特定事件触发时自动执行的脚本，可拦截或追加操作。

Plugins

插件

三者的打包封装——一键分享完整配置给团队，或换电脑时一键恢复。

类比：开卷考试 → MCP = 你有没有把参考书带进考场；Skills = 你知不知道怎么快速翻书找答案。两者缺一不可。

三大必装 MCP

MCP作用安装命令
filesystem 读写电脑任意位置文件，不受当前文件夹限制 claude mcp add filesystem ...
brave-search 实时搜索网页，配合 Loop 做资讯采集 claude mcp add brave-search ...
lark-mcp（飞书） 回消息、发通知、定会议、安排日程 claude mcp add lark-mcp ...
查看已安装的 MCP 和 Skills
/mcp # 查看 MCP 列表 /skills # 查看 Skills 列表
Skills 越精越好，不是越多越好。Skills 过多 = 模型选择困难症，不知道何时该调用哪个，执行效果反而变差。建议在使用中发现高频重复工作流后再针对性创建。

Skill 标准文件结构

skill-template.md
# Skill 名称 ## 触发条件 什么情况下使用这个 Skill ## 执行步骤 1. 第一步做什么 2. 第二步做什么 ## 输出规范 - 格式要求 - 保存位置：D:\文件夹\文件名规则 ## 禁止事项 - 不能做什么 - 遇到什么情况时停止并告知我

Hooks——自动触发钩子

两类 Hooks

类型触发时机典型用途
Pre-tool Hook工具调用前拦截敏感文件访问、参数检查
Post-tool Hook工具调用后自动 git push、发送通知

两个真实案例

案例 1：隐私保护 Hook

创建指令

四大隐藏命令——精华中的精华

/simplify — 三 Agent 并行代码审查

Claude Code 内
/simplify

执行后派出三个子 Agent 并行检查变更代码：

Agent检查内容
Agent ①重复造轮子、冗余代码
Agent ②命名规范、可读性、代码质量
Agent ③性能隐患、内存泄露、重复计算
前提：项目需有 .git 文件夹。它只审查 git diff 的变更部分，不扫描整个项目。建议每次提交前运行一次。

/rewind — 时光倒流

Claude Code 内
/rewind # 快捷键：Esc + Esc
选项作用
Restore conversation在旧检查点开新分支继续，原对话不变。最常用。
Summarize from here压缩中间对话成摘要再继续，释放上下文空间
Nevermind取消操作
注意：只能恢复 Claude 的文件编辑操作，无法撤销已执行的 Bash 命令（如 rm / mv）。

/insights — 协作习惯体检

Claude Code 内
/insights

读取所有历史对话，生成报告：你在哪里花时间最多、在哪类问题上最容易卡住、常见决策模式、token 消耗分析。建议每两周运行一次，根据结果优化 CLAUDE.md 和 Skills。

/loop — 定时循环任务

Claude Code 内
# 基础用法 /loop interval=1h 帮我查询当前时间 # 与 Skill 结合（杀手锏） /loop interval=24h time=08:00 执行「ai-news」Skill
限制说明
有效期最长 3 天，到期自动停止
运行条件Claude Code 终端必须保持开启
最大任务数每个会话最多 50 个

Plugins——配置打包带走

Plugin = Hooks + Skills + MCP 的封装包。适合团队共享配置、换电脑一键恢复环境。

官方推荐：Codex Plugin

终端 + Claude Code
# 终端安装 npm install -g claude-plugin-cc # 进入 Claude Code 后 /setup codex

安装后，Claude Code 可直接调用 Codex，两个 AI 自动协作：你发任务 → Claude Code 拆解 → 子任务交给 Codex 执行 → 结果回传 → 继续下一步。全程无需人工传递上下文。

12

Handoff 文件——跨对话记忆移植

问题根源：Claude Code 上下文窗口是一次性的，新对话所有"共识"清零。对话超过 20-30 轮后，autocompact 会压缩上下文，早期架构约束被遗忘，莫名 Bug 越来越多。

何时生成

场景口诀
对话超过 20-30 轮"话多了，写纸条"
完成阶段性任务后"一阶段，一交接"
Claude 开始重复犯错"重复错，快交接"
跨天继续工作前"跨天班，必交接"

标准模板

handoff.md
# Handoff - [日期] [任务名] ## 当前状态 - 整体进度：XX% - 最后完成的事：... ## 已验证有效的方案 - 用了 XXX 方法解决了 YYY 问题 - 关键参数：... ## ❌ 已排除的方案（最重要！） - 方案 A：试过，会导致内存泄漏 - 方案 B：与当前版本不兼容 ## 下一步 1. 待解决：... 2. 待验证：... ## 关键约束（不能破坏） - 不能动 XX 文件 - 必须兼容 YY 版本 ## 相关文件路径 - 主文件：src/... - 配置：config/...
「已排除方案」是价值最高的一栏。没有它，新对话的 Claude 会重新"发明"你们已否定过的路径，你又要花时间重新解释"这个我们试过，不行"——这正是上下文污染最浪费时间的地方。

加载到新对话

新对话第一句话
请先读取 handoff.md，了解我们的进度和约束， 然后复述一遍你的理解，确认后我们继续。

让它复述理解是验证加载有效的关键步骤，不要跳过。

CLAUDE.md vs Handoff 的分工

CLAUDE.mdHandoff 文件
内容永久性项目规则临时性任务进度
更新频率很少改每次任务后更新
生效方式自动加载手动加载
类比公司规章制度项目交接班日志

节省 Token——控制成本

最大的隐性消耗：工具调用噪声

每次调用工具（运行测试、搜索文件）会产生大量输出，全部塞进上下文。模型只需知道"成功/失败"，那几百行日志是纯噪声。

解决方案：安装 RTK

终端
curl -fsSL https://get.rtk.dev/install.sh | sh rtk init # 自动创建 Pre-tool Hook，拦截并压缩输出
命令原始 TokenRTK 压缩后节省
git status3,00060080%
cargo test25,0002,50090%

其他节省习惯

习惯效果
CLAUDE.md 保持精简每次对话少加载几千 token
复杂子任务交给独立 Agent主对话不被中间过程撑大
长对话用 Handoff 开新对话避免多次压缩损耗质量
Plan Mode 先规划再执行减少走弯路浪费的 token
优先用 CLI 工具而非 MCPCLI 不注入工具定义，更省

Step 1：创建保存文件夹

对话指令

对话指令（完整）

思维转变 & 四步入门路径

核心思维转变

❌ 旧思维：问 AI
每次从零解释需求，等待单次回答，手动执行结果。
✅ 新思维：指挥 AI
一次配置 Skill，反复触发执行，AI 自主汇报结果。

你的四步入门路径

1
运行 /init，生成 CLAUDE.md

进入任意工作文件夹，输入 /init，让 Claude 了解项目结构，生成初始配置文件。

2
编辑 CLAUDE.md，写下"家规"

写 3-5 条基本规则（禁止删除文件、回复用中文、禁止危险命令等），其余详细内容引用外部文件。

3
安装 filesystem MCP

让 Claude Code 获得操作你电脑任意位置文件的能力，告诉它你的常用目录路径。

4
完成第一个真实任务

整理 Downloads 文件夹 + 设置 Loop 定时。把教程知识变成肌肉记忆。
