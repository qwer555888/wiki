MODULE 01

三种运行模式

核心理解

三种模式控制的是 Claude Code 的执行权限边界。不是说哪种好哪种坏，而是不同场景下用不同模式可以最大化效率同时控制风险。

关键区分：文件操作权限和终端命令权限是分开的两套权限体系。自动模式只豁免文件操作，终端命令始终需要确认，除非启动时加了危险参数。

三种模式对照表

底部显示模式名文件操作终端命令适用场景
? for shortcuts默认模式每次询问每次询问学习阶段、不熟悉的项目
accept edits on自动模式自动通过仍然询问日常开发、信任的项目
plan mode规划模式只读，不执行不执行讨论方案、复杂任务前期

完全跳过所有确认

PowerShell / 终端
claude --dangerously-skip-permissions # 底部显示：bypass permissions on # 文件操作 + 终端命令 全部自动通过
危险模式真正的风险：Claude Code 理论上拥有和你一样的终端权限，可以执行 rm -rf、修改系统文件、安装任意程序。虽然 AI 主动破坏电脑的概率极低，但用前确认你的 CLAUDE.md 里写了禁止危险命令的规则。

三个真实项目场景

01
个人博客网站开发
推荐：自动模式

场景：你要做一个静态博客，需要创建几十个组件文件、样式文件、配置文件。

为什么用自动模式

创建组件文件是安全的低风险操作，每次都确认会打断思路。用 accept edits on 模式让 Claude Code 自由创建修改文件，遇到 npm install 等终端命令时系统还会询问，保留了最后一道防线。

操作流程
1. 启动：claude 2. Shift+Tab 切换到 accept edits on 模式 3. 输入：帮我做一个博客网站，使用 React + Tailwind， 包含首页、文章列表、文章详情、关于我页面
02
重构旧项目架构
推荐：先 Plan 后自动

场景：把一个几千行的 jQuery 老项目重构成 React，涉及大量文件的删除、新建、修改。

为什么先用规划模式

重构是高风险操作，先进 Plan Mode 让 AI 列出完整计划，你确认架构思路没问题后再放行执行。一旦发现计划有问题，在规划阶段修正成本为零；执行后再发现就要靠回滚了。

操作流程
1. Shift+Tab 进入 plan mode 2. 输入：将当前 jQuery 项目重构为 React + TypeScript， 保留所有现有功能，UI 风格保持一致 （Shift+回车换行继续输入） 3. 审核计划，确认架构方向正确 4. 选择「执行计划并进入自动模式」 5. 等待执行完成
03
自动化数据处理脚本
推荐：危险模式

场景：需要写一个 Python 脚本，批量处理 500 个 Excel 文件，安装依赖、创建目录、移动文件都是必要操作。

为什么用危险模式

这类任务涉及大量终端命令（pip install、mkdir、mv 等），每个都确认效率极低。在你已经明确知道任务范围的前提下，用危险模式让 Claude Code 全自动完成。前提：在 CLAUDE.md 里写明「禁止删除原始数据文件」。

操作流程
1. 启动：claude --dangerously-skip-permissions 2. 输入：帮我写一个 Python 脚本，批量读取 D:\data\ 目录下所有 Excel 文件，提取第一列数据， 合并成一个总表保存到 D:\output\result.xlsx 注意：不得删除或修改原始文件

MODULE 02

后台任务管理

核心理解

Claude Code 是单线程的——同一时间只能做一件事。当你启动一个服务（如 npm run dev），它会一直占据 Claude Code 的处理能力，让你无法继续对话。

后台任务解决的就是这个问题：把长期运行的进程"挂起"到后台，Claude Code 恢复空闲，你可以继续下指令。服务仍在运行，只是不再占前台。

核心命令

Claude Code 内操作
# 把当前运行的服务放到后台 Ctrl + B # 查看所有后台任务 /tasks # 在 /tasks 界面里关闭某个任务 按 K # 退出 /tasks 界面 ESC

高频使用场景

场景：开发 React 项目，需要同时运行开发服务器，并且持续让 Claude Code 修改代码，实时看效果。

让 Claude 写代码
→
npm run dev 启动服务

Ctrl+B 放后台
→
继续让 Claude 改代码

浏览器实时看效果

Claude Code 写完代码后，输入：!npm run dev 服务启动后按 Ctrl+B 放到后台 继续输入：在右上角加一个深色模式切换按钮 浏览器里 localhost:3000 已经实时更新了

长任务管理

MODULE 03

回滚功能

核心理解

Claude Code 每次执行操作后都会自动保存一个检查点（Checkpoint）。回滚就是回到某个检查点的状态。

关键局限：只能回滚 Claude Code 自己写入的文件。由终端命令产生的内容（npm install 的 node_modules、mkdir 创建的目录）无法回滚。这是为什么生产项目必须用 Git 做版本控制的原因。

操作方式

Claude Code 内
# 方式一：最快（推荐） 按两下 ESC # 方式二：命令方式 /rewind

四个回滚选项的含义

选项含义适用情况
恢复代码 + 对话完全回到过去，代码和聊天记录都还原做错了，完全重来
只恢复对话聊天记录还原，但代码保留当前状态想重新规划思路但保留代码
只恢复代码代码还原，但对话记录不变代码写错了但对话有参考价值
放弃回滚取消操作误触了 ESC

三个真实项目场景

01
UI 改版改错了
最常见回滚场景

场景：让 Claude Code 给网站换了个新主题，改完发现原来的更好看，想退回去。

操作步骤
1. 按两下 ESC 进入回滚界面 2. 找到「换主题」之前的那个检查点 3. 选择「恢复代码 + 对话」 4. 验证：浏览器刷新，原来的主题回来了 ✓ 注意：如果期间运行过 npm install 安装了新包， 那些包不会被回滚，需要手动 npm uninstall
02
功能方向走偏，重新来过
方向性回滚

场景：让 Claude Code 实现一个搜索功能，写了一半发现实现思路不对，想换一种方案。

操作步骤
1. 按两下 ESC 回到搜索功能开始前的检查点 2. 选择「恢复代码 + 对话」 3. 重新输入需求，这次明确指定实现方式： 帮我实现搜索功能，使用 Fuse.js 模糊搜索库， 不要用后端 API，直接在前端本地搜索
回滚 + Plan Mode 组合用法：回滚后先进 Plan Mode，把新思路列出计划确认后再执行，避免再次走偏。
03
为什么生产项目一定要用 Git
重要认知

场景：Claude Code 回滚的局限性演示——展示什么情况下回滚会"失效"。

❌ Claude Code 回滚的局限

只能回滚自己写的文件
npm install 产生的包回滚不了
mkdir 创建的目录回滚不了
数据库改动回滚不了
检查点只保留 30 天

✅ Git 的优势

精确追踪每一行代码变化
可以 diff 对比任意版本
branch 可以同时保留多个方向
永久保存不会过期
团队协作必备

让 Claude Code 帮你初始化 Git
帮我初始化 git 仓库，创建 .gitignore 文件 （忽略 node_modules、.env、dist 等）， 然后做第一次 commit，信息写「初始化项目」

MODULE 04

多模态传图给 Claude

核心理解

Claude Code 具备视觉能力，可以"看懂"图片。你可以传入截图、设计稿、原型图、错误截图，让它理解视觉内容后做出对应的修改。

传图方式有两种，效果一样，选择方便的那种就行。特别注意：macOS 里粘贴图片也必须用 Ctrl+V，不是 Cmd+V。

两种传图方式

操作方式
# 方式一：拖拽 直接把图片文件拖到 Claude Code 输入框 # 方式二：粘贴（macOS 也是 Ctrl+V！） 复制图片文件 → 在 Claude Code 输入框按 Ctrl+V # 确认图片已接收 输入框里会显示 [image] 标记

三个真实项目场景

01
截图仿写竞品页面
最常见用法

场景：看到一个竞品的漂亮登录页，截图后让 Claude Code 仿写一个类似风格的。

操作流程
1. 对竞品登录页截图（macOS: Cmd+Shift+4） 2. 把截图拖到 Claude Code 输入框 3. 输入：参考这张截图的风格和布局， 帮我做一个登录页面（使用 HTML + CSS）， 不要照抄内容，只参考视觉风格 4. 回车执行
准确度说明：传图还原通常有 70-80% 的相似度，字体、精确间距、特殊效果往往会有偏差。追求高精度还原请用 Figma MCP（见 Module 05）。
02
传错误截图修 Bug
调试神器

场景：浏览器控制台出现了一堆红色报错，不想手动复制文字，直接截图发给 Claude Code。

操作流程
1. 打开浏览器控制台（F12） 2. 对报错内容截图 3. 拖入 Claude Code 输入框 4. 输入：看一下这个报错，帮我找到原因并修复 5. Claude Code 会识别报错内容， 定位到对应代码，自动修复
03
传手绘草图转代码
创意工作流

场景：在纸上手绘了一个 App 界面草图，拍照后让 Claude Code 把草图变成真实代码。

操作流程
1. 手机拍摄草图照片 2. 传到电脑（AirDrop / 微信传文件等） 3. 把图片拖入 Claude Code 4. 输入：这是我手绘的 App 界面草图， 请按照这个布局用 HTML + CSS 实现， 风格参考 iOS 设计规范，使用现代简洁风格 # Claude Code 会识别草图中的： # - 按钮位置和文字 # - 输入框 # - 列表结构 # - 大致布局比例

MODULE 05

Figma MCP精准还原设计稿

核心理解

普通传图只能让 Claude Code "看到"设计稿的视觉外观，无法获取精确数据。Figma MCP 让 Claude Code 直接从 Figma 读取完整设计数据：每个元素的精确尺寸、间距、字体大小、颜色值、圆角数值……

结果就是：还原度从 70% 提升到 95%+，字体、间距、颜色一个不差。

安装与配置流程

终端
# 第一步：按 Figma 官方文档安装 MCP Server # （具体命令以官网为准） # 第二步：重启 Claude Code 后授权 /mcp → 选择 figma → authenticate # 会弹出浏览器，点同意授权 # 第三步：验证安装成功 /mcp → 选择 figma → view tools # 能看到工具列表说明成功

使用方式

Claude Code 对话
修改当前页面，使它与 Figma 设计稿保持一致 [粘贴 Figma 设计稿链接] # Claude Code 会自动调用： # 1. getDesignContext：获取所有组件的设计数据 # 2. getScreenshot：获取视觉截图 # 然后对比现有代码，精准修改

三个真实项目场景

01
设计稿转完整网页
最核心用法

场景：设计师在 Figma 做好了整个网站首页设计稿，让 Claude Code 把设计稿变成真实代码。

操作流程
1. 在 Figma 里选中要实现的 Frame 2. 右键 → Copy link to selection 3. 回到 Claude Code 输入： 请根据这个 Figma 设计稿， 用 HTML + CSS 实现完整的首页， 要求像素级还原，字体间距颜色与设计稿一致 [粘贴链接] 4. Claude Code 调用 MCP 获取设计数据 5. 自动生成高度还原的代码
02
组件库精准还原
团队协作场景

场景：设计师做了一套完整的 UI 组件库（按钮、输入框、卡片等），让 Claude Code 转成 React 组件。

操作流程
请读取这个 Figma 组件库， 把所有基础组件（Button、Input、Card、Badge） 转成 React + TypeScript 组件， 要求： - Props 设计合理（size, variant, disabled 等） - 样式使用 CSS Modules - 颜色值使用 CSS 变量 [粘贴 Figma 组件库链接]
03
设计评审对比检查
质量把控场景

场景：开发完成后，让 Claude Code 自动对比现有代码和 Figma 设计稿，找出不一致的地方。

操作流程
请对比当前的 index.html 和这个 Figma 设计稿， 找出所有视觉上不一致的地方， 按优先级（高/中/低）列出清单， 然后自动修复「高优先级」的问题 [粘贴 Figma 链接]
这个流程相当于一个自动化的设计走查工具，能发现开发过程中细微的偏差（如间距差了 2px、颜色色号写错了）。

MODULE 06

上下文管理/compact 与 /clear

核心理解

上下文窗口就像一张纸，Claude Code 把对话过程、代码、工具调用结果都写在这张纸上。纸写满了，早期的内容会被自动压缩甚至遗忘，导致 AI 开始犯之前犯过的错误。

主动管理上下文比等它自动压缩要好得多：你可以控制保留什么、丢弃什么。

两个命令的区别

/compact — 智能压缩

把上下文压缩成摘要。
保留关键信息，丢弃冗余细节。
就像把 100 页对话提炼成 5 页摘要。
适合：同一任务继续推进时。

/clear — 完全清空

把所有上下文全部删掉。
Claude Code 变成"失忆"状态。
像打开全新的对话窗口。
适合：完全换一个新任务时。

Claude Code 内
# 压缩上下文（保留摘要） /compact # 带策略的压缩（更精准） /compact 重点保留：用户的功能需求、已确定的架构决策、待解决的 Bug # 查看当前上下文内容 Ctrl + O # 完全清空 /clear

三个真实项目场景

01
长时间开发项目中途压缩
最常用场景

场景：一个项目做了 3 小时，对话里有大量的调试日志、临时代码、MCP 调用结果，Claude Code 开始变慢，偶尔犯之前犯过的错误。

压缩策略
# 先生成 handoff 文件，记录关键进度 请生成一份 handoff.md，记录： 当前完成了什么、下一步要做什么、 已确定的技术方案、已排除的方案 # 然后压缩上下文 /compact 重点保留：项目架构决策、待解决的 bug 列表、用户的功能需求 # 压缩后按 Ctrl+O 检查摘要是否完整 # 确认后继续开发
02
MCP 调用后清理噪声
提升性能

场景：用 Figma MCP 或搜索 MCP 后，大量的 JSON 数据塞满了上下文，后续对话变慢。

压缩策略
Figma MCP 调用完成，代码写好后： /compact 重点保留：已完成的代码结构、 用户的设计还原需求、待修改的细节列表 # MCP 调用产生的大量 JSON 数据会被丢弃 # 上下文变小，后续响应速度恢复正常
03
切换到完全不相关的新任务
清空场景

场景：上午在做电商网站，下午要开始做完全不相关的数据分析脚本，之前的上下文完全没用。

操作流程
# 先保存上午的进度（重要！） 请生成 ecommerce-handoff.md 记录当前进度 # 清空上下文 /clear # 现在开始新任务，上下文干干净净 帮我写一个 Python 脚本， 分析 CSV 文件中的销售数据， 生成月度趋势图表（使用 matplotlib） # 下午恢复电商网站开发： # claude -c 恢复会话 → 读取 handoff 文件 → 继续

MODULE 07

会话恢复

核心理解

每次关掉终端再重新打开 Claude Code，默认是全新状态，之前的对话消失了。但 Claude Code 会保存历史会话，你可以随时恢复到之前的任意一个会话继续工作。

两种恢复方式

终端
# 方式一：自动恢复上一次会话（最常用） claude -c # 方式二：列出所有历史会话，手动选择 # 进入 Claude Code 后输入： /resume # 会列出所有历史会话，选择对应的回车

三个真实项目场景

01
每天继续昨天的开发
日常工作流

场景：昨天做了一半的功能，今天早上继续。

最佳工作流
# 昨天下班前：生成 handoff 文件 请生成 handoff.md，记录今天的进度 # 今天早上： cd my-project claude -c # 恢复昨天的会话 # 进入后： 请先读取 handoff.md，了解昨天的进度， 然后告诉我你的理解，确认后我们继续
02
安装 MCP 后必须重启
常见操作

场景：安装了新的 MCP Server，需要重启 Claude Code 让它生效，但不想丢掉当前对话。

操作流程
# 安装 MCP 后，退出当前 Claude Code Ctrl + C 或输入 /exit # 重新启动并恢复对话 claude -c # 验证 MCP 已加载 /mcp # 看到新安装的 MCP 出现在列表里 = 成功
03
多项目切换管理
项目管理

场景：同时维护三个项目，需要在不同项目的会话之间切换。

操作流程

MODULE 08

CLAUDE.md项目宪法

核心理解

CLAUDE.md 是 Claude Code 每次启动都会自动读取的文件，相当于给 AI 的永久指令。无论过了多久、换了多少次会话，只要进入这个项目，这些规则始终生效。

有两个级别：项目级（只对当前项目生效）和用户级（对当前用户所有项目生效）。两者可以同时存在，互相叠加。

操作命令

Claude Code 内
# 自动生成 CLAUDE.md /init # 快速打开 CLAUDE.md（不用去文件夹找） /memory # 选择「项目级别」或「用户级别」 # 修改完必须重启 Claude Code 才生效

三个真实项目场景

01
前端团队项目规范
团队协作

场景：5 人前端团队，希望所有人用 Claude Code 写出来的代码风格一致，不用每次都重新解释规范。

CLAUDE.md 示例
# 前端项目开发规范 ## 技术栈 - React 18 + TypeScript 5 - Tailwind CSS（不使用 CSS Modules） - Zustand 状态管理（不使用 Redux） - React Query 数据请求 ## 代码规范 - 组件使用函数式，禁止 Class 组件 - Props 必须定义 TypeScript 接口 - 文件命名：组件用 PascalCase，工具函数用 camelCase - 禁止使用 any 类型 ## 禁止操作 - 禁止修改 package.json 的 dependencies 版本 - 禁止删除 src/types/ 目录下的文件 - 提交前必须通过 ESLint 检查 ## 详细规范 如需了解组件写法规范，请查阅 docs/component-guide.md
02
个人全局偏好设置
用户级 CLAUDE.md

场景：不管在哪个项目，你都希望 Claude Code 遵循同样的沟通方式和一些个人偏好。

用户级 CLAUDE.md（~/.claude/CLAUDE.md）
# 我的全局偏好 ## 沟通方式 - 所有回复使用中文 - 解释技术概念时先说结论，再说原因 - 代码注释也用中文 ## 操作习惯 - 修改文件前先告诉我打算做什么 - 遇到多种实现方案时，列出优劣后推荐最优方案 - 禁止在任何项目中执行 rm -rf ## 格式偏好 - 代码使用 2 空格缩进 - 字符串优先用单引号 - 语句结尾不加分号（JavaScript）
03
AI 日报项目记忆
项目记忆

场景：你的 AI 日报自动化项目，每次进入都要让 Claude Code 了解这个项目是干什么的。

CLAUDE.md 示例
# AI 日报自动化项目 ## 项目说明 这是一个自动采集 AI 行业新闻的系统， 每天早上 8 点自动运行，生成日报文件。 ## 目录结构 - D:\AI日报\ 存放每日日报文件 - D:\AI日报\templates\ 存放日报模板 - skills/ai-news.md 是核心 Skill 文件 ## 关键规则 - 日报文件命名格式：YYYY-MM-DD.md - 禁止修改 templates 目录下的文件 - 搜索关键词包括：大模型、AI产品、机器学习、AGI ## 常用命令 - 手动触发：说「抓取AI新闻」 - 查看定时任务：/tasks

MODULE 09

Hooks自动触发脚本

核心理解

Hooks 是"事件监听器"——当某个特定事件发生时，自动执行你预设的脚本。你不需要记得手动执行这些步骤，系统会在正确的时机自动触发。

两大类：Pre-tool（操作前）用于拦截和检查；Post-tool（操作后）用于追加自动化动作。

创建 Hook

Claude Code 内
# 进入 Hook 配置界面 /hook # 选择触发时机： # PostToolUse - 工具执行后 # PreToolUse - 工具执行前 # Notification - 发送通知时 # 选择触发工具： # Write/Edit - 创建或编辑文件后 # Bash - 执行终端命令后 # Read - 读取文件前/后

三个真实项目场景

01
代码写完自动格式化
最常用 Hook

场景：Claude Code 写完代码，自动用 Prettier 格式化，保证代码风格统一，不用每次手动运行。

Hook 配置
# 触发时机：PostToolUse # 触发工具：Write 或 Edit # 执行命令： echo "$CLAUDE_TOOL_INPUT" | jq -r '.file_path' | xargs prettier --write # 效果： # Claude Code 写完任何文件 → 自动 Prettier 格式化 # 你看到的代码永远是格式化好的
前提：需要先安装 prettier（npm install -g prettier）和 jq（解析 JSON 的命令行工具）。
02
读取文件前拦截敏感信息
安全防护

场景：项目里有 .env 文件存放 API Key，不希望 Claude Code 读取这些敏感信息并发送给 AI。

Hook 描述（告诉 Claude Code 创建）
帮我创建一个 Pre-tool Hook： 触发时机：读取文件前（PreToolUse，工具：Read） 逻辑：检查即将读取的文件路径， 如果路径包含以下关键词则拦截并提示我： .env / .secret / password / private_key / id_rsa 拦截后输出：「⚠️ 即将读取敏感文件：[路径]，是否确认？」
03
代码修改后自动提交 Git
自动化工作流

场景：Claude Code 每次修改代码后，自动执行 git add + git commit，保留完整的修改记录。

Hook 描述
帮我创建一个 Post-tool Hook： 触发时机：文件编辑完成后（PostToolUse，工具：Write/Edit） 逻辑：自动执行以下命令： git add [被修改的文件路径] git commit -m "auto: Claude Code 修改 [文件名] [当前时间]" 只在有 .git 目录的项目里生效， 如果 git commit 失败不要报错，静默处理
注意：这个 Hook 会产生大量小 commit，适合开发阶段的探索性工作。正式提交前记得用 git rebase -i 合并 commit。

MODULE 10

Agent Skill vs Sub-agent

核心理解——最重要的区别

Agent Skill：助手坐在你旁边干活。所有过程、中间结果、思考过程全都出现在你们共同的对话里，你能看到一切，但上下文会越来越大。

Sub-agent：你派人出去办事。他拿着任务单去了另一个房间，做什么、遇到什么、怎么解决的你不知道。他办完了只带回一份结果汇报。你们共同的对话保持干净。

对比表

Agent SkillSub-agent
上下文共享主对话，过程全部可见完全独立，主对话只收结果
Token 消耗高（中间过程都计入）低（主对话只有最终汇报）
适合任务轻量、与上下文关联紧密重型、独立性强、大量中间操作
典型例子写每日总结、生成报告扫描整个代码库、批量处理文件
创建方式在 ~/.claude/skills/ 放 .md 文件/agent → Create new agent

三个真实项目场景

01
每日开发总结（用 Agent Skill）
Skill 适用场景

场景：每天下班前，根据今天的开发对话内容，自动生成一份结构化的日报。

为什么用 Skill 而不是 Sub-agent：总结任务需要读取今天的对话内容，必须共享上下文，且生成的总结文字量不大，不会严重污染上下文。

skill.md 文件内容
--- name: daily-report description: 当用户要求写每日总结时调用 --- # 每日开发总结生成规范 ## 输出格式 # 开发日报 - [今天日期] ## 今日完成 （列出今天实现的功能，每条一行） ## 遇到的问题与解决方案 （列出遇到的 Bug 和解决思路） ## 明日计划 （列出明天要继续做的事） ## 技术备忘 （记录今天学到的技术点） ## 规则 - 语言：中文 - 保存到：D:\开发日志\[日期].md - 内容从今天的对话中提取，不要编造
02
全项目代码审查（用 Sub-agent）
Sub-agent 适用场景

场景：项目有几万行代码，需要做一次全面的代码审查，找出所有潜在问题。

为什么用 Sub-agent 而不是 Skill：扫描几万行代码会产生海量中间内容，如果在主对话里做，上下文会爆掉，主对话之后什么都做不了。

Sub-agent 描述文件
--- name: code-reviewer description: 全项目代码审查，只使用只读工具 model: inherit color: green --- # 代码审查专家 你是一个严格的代码审查专家。 当被调用时，你需要： 1. 扫描整个 src/ 目录下的所有文件 2. 检查以下问题： - 未处理的 Promise rejected - 可能的内存泄漏（事件监听未移除） - 重复代码（相似度超过 80%） - 硬编码的密钥或敏感信息 - TypeScript any 类型的滥用 3. 输出格式： ## 审查报告 - [日期] ### 高危问题（需立即修复） ### 中优问题（建议修复） ### 低优问题（可以优化） 每个问题注明：文件路径、行号、问题描述、修复建议 4. 只读取文件，不修改任何内容
调用方式：在主对话里说「帮我做一次全项目代码审查」，Claude Code 会自动识别并启动 code-reviewer Sub-agent。Sub-agent 在独立环境里扫描所有文件，最后只把审查报告发回主对话。

MODULE 11

Plugin一键安装整套能力

核心理解

Plugin 是 Skills + Sub-agents + Hooks + MCP 的打包安装包。就像手机的 App——你不需要知道里面怎么实现的，安装完直接用。

两种用途：①安装别人做好的插件获得新能力；②把自己的配置打包成插件分享给团队，一键同步所有人的开发环境。

操作命令

Claude Code 内
# 进入插件管理器 /plugin # 三个选项： # Discover - 发现新插件 # Installed - 查看已安装插件 # Featured - 推荐插件 # 安装后验证（以 frontend-design 为例） /skills # 看到对应 Skill 出现 = 安装成功

三个真实项目场景

01
安装 frontend-design 提升 UI 质量
官方推荐插件

场景：Claude Code 做出来的前端界面总是千篇一律的深紫色渐变风格，安装这个插件让它的 UI 设计能力升级。

❌ 没有插件的效果

深紫色背景
白色圆角卡片
Inter 字体
每个项目长一样

✅ 安装后的效果

色彩更协调
排版更有层次
符合现代审美
每个项目有自己的风格

使用方式
# 安装插件 /plugin → Discover → 找到 frontend-design → 安装 # 使用时明确说明要使用这个插件 按照 frontend-design 的要求， 帮我做一个待办事项 App，使用 HTML 实现 # Claude Code 会先读取 frontend-design Skill， # 获取 Anthropic 官方沉淀的 UI 设计规范， # 然后按照规范写代码
02
团队共享开发环境插件
团队协作

场景：你花了一周时间配好了一套完美的开发环境（代码格式化 Hook + 日报 Skill + 代码审查 Sub-agent），想让团队所有人一键获得同样的配置。

打包成插件
帮我把当前项目的所有配置打包成一个插件： - 名称：team-dev-kit - 包含： ① code-formatter Hook（写完代码自动格式化） ② daily-report Skill（每日总结） ③ code-reviewer Sub-agent（代码审查） - 保存到项目根目录，让团队成员可以一键安装 # 团队其他成员只需要： # /plugin → Install → 选择 team-dev-kit → 回车 # 所有配置自动到位
03
换电脑一键恢复环境
个人效率

场景：换了新电脑，不想一个一个重新配置 Hooks、Skills、Sub-agents。

备份与恢复流程
# 旧电脑：把所有配置打包成插件 帮我把我的所有 Skills、Hooks、Sub-agents 打包成一个叫「my-personal-kit」的插件， 保存到 ~/my-claude-kit/ 目录 # 把 ~/my-claude-kit/ 目录同步到新电脑 （通过网盘/U盘/Git 均可） # 新电脑：一键安装 /plugin → Install from local path → 选择目录 # 所有配置恢复完毕
Plugin 相当于你的 Claude Code 「存档文件」，建议定期更新备份。尤其是在配置了很多自定义内容之后，备份一份以防丢失。
